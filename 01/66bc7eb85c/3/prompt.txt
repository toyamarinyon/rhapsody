Review feedback from parent: the current implementation is not yet acceptable. Please address these concrete issues in the same workspace without reverting unrelated changes.

Critical issues:
1) User explicitly wants buildable comments too. Current `postIntakeComment` only runs for non-buildable. Change to comment for buildable/ask_human/blocked/skip, with dedupe.
2) The requirement is `blocked_by` set AND referenced issue is not closed -> skip. Current implementation treats any blockedBy string as blocked and does not model closed/open state. Add a minimal structured blocker type (string compatibility ok for tests) with state, and only block open/unknown blockers. Closed blockers should proceed to classifier/buildable path.
3) Classifier schema should not include `blocked_by` as an LLM decision if blocked_by is deterministic. Use outcomes `buildable | ask_human | skip` for LLM, and worker outcomes can include `blocked` (or `blocked_by`, but be consistent). The final product should have a skip/block decision for deterministic blockers.
4) `runCodexIntakeClassification` currently calls `runCodexExec(command)` with a built command object, but `runCodexExec` expects CodexCliOptions and builds its own command. Fix this.
5) Zod business validation is too weak. Need buildable requires implementation plan/comment, ask_human requires question/comment, skip requires reason/comment. Current schema only has reason/comment optional-ish.
6) There is no real comment dedupe. Fresh decision dedupe prevents rerun only when existingDecisions are passed, but comment artifact dedupe should be explicit enough for repeated scheduler ticks with graph decisions. At minimum, if fresh decision exists return based on it and do not comment; if creating new decision create one comment artifact. Prefer metadata fingerprint.
7) package.json dependency versions need to be installable and pnpm-lock.yaml must be updated. Use known compatible zod + zod-to-json-schema versions, or avoid zod-to-json-schema by hand-generating JSON schema from the Zod source if dependency friction is high.
8) Keep raw command strings out of evidence/logs if they can include paths only fine, but no prompt/token. Ensure raw classifier output is not stored unless sanitized.
9) Tests should cover closed blocker proceeds, open blocker skips, buildable posts a comment, invalid primary -> healing -> success, primary+healing invalid -> fallback ask_human.
