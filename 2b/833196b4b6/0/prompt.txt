実装をお願いします。ユーザーの明示的依頼により、軽微な OAuth refresh 修正をあなたに委譲します。こちらはレビュー・PR・merge・動作確認を担当します。既存変更を勝手に revert しないでください。

背景:
- backend-probe は production DB の access_token で ChatGPT backend-api に 200 OK。
- refresh-probe は production DB の refresh_token で auth.openai.com/oauth/token に 200 OK。persist:true で DB 更新も成功。
- それでも sandbox Codex intake は `Your access token could not be refreshed` で失敗。
- 差分として、proxy の `handleAuthTokenExchange` は sandbox request body の `client_id` を `readOAuthClientId(body) ?? CHATGPT_OAUTH_CLIENT_ID` で優先している。一方、refresh-probe は固定 `CHATGPT_OAUTH_CLIENT_ID` で成功している。

目的:
- ChatGPT proxy の OAuth refresh では、sandbox から来た `client_id` を信用せず、mediator が管理する固定 `CHATGPT_OAUTH_CLIENT_ID` を常に使うようにしてください。

対象:
- `app/api/internal/codex-chatgpt-proxy/[...path]/route.ts`
- 関連 test が必要なら追加/更新してください。

期待変更:
- `handleAuthTokenExchange` の upstream body を:
  - `client_id: CHATGPT_OAUTH_CLIENT_ID`
  - `grant_type: "refresh_token"`
  - `refresh_token: state.refreshToken`
  にする。
- `readOAuthClientId` が不要になれば削除して OK。
- テストで、sandbox body に別 client_id が入っていても upstream request は `CHATGPT_OAUTH_CLIENT_ID` を使うことを保証してください。既存 route tests に足すか、新規で最小テストを作ってください。
- token 値をレスポンス/log/test snapshot に出さない方針は維持。
