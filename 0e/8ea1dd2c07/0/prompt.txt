Review feedback: good first pass, but there are several must-fix correctness/safety issues before this is acceptable. Please patch them.

1. Fresh duplicate retry can loop forever after repair_failed.
- In scheduler fresh-duplicate path, it reuses the latest repair_allowed decision and calls executor without recomputing budgets.
- Since repair_failed is not terminal and active run ends, the next tick will reuse the same repair_allowed forever, ignoring the growing repair_failed count.
- Fix: on fresh duplicate, do not directly execute old repair_allowed when there are subsequent repair_failed attempts. Re-run runRepairerPlanner with existing graph decisions/checkSummary so budgets are recomputed, then execute only if the new plan outcome is repair_allowed. Alternatively, build a helper that detects failed retries and recomputes. Terminal repair_applied/repair_noop should still skip. Unexecuted old repair_allowed with no execution attempts may be executed.
- Add/adjust a test that two prior repair_failed decisions at the same PR/head/fingerprint cause no executor call / repair_blocked on the duplicate path.

2. format-repair-wrapper lacks diff guard and can commit unrelated files.
- It currently runs npm install, then formatter, then git add . and commits every changed file. That can include package-lock.json or any install side effect.
- Fix: implement a real diff guard. The wrapper metadata should include allowedChangedFiles derived from the repair failure evidence/fingerprint/check output when available. For the first pass, if allowedChangedFiles is empty, fail closed (repair_failed) rather than format the whole repo blindly.
- After formatter, compute changed files and ensure every changed file is in allowedChangedFiles. If any unexpected file changed, emit repair_failed and do not commit/push.
- Stage only allowed changed files, not git add .
- Add wrapper or executor tests for unexpected changed files causing repair_failed/no push if practical, or unit-test a helper exported/duplicated in TS if wrapper direct test is awkward.

3. package manager selection is unsafe for this repo.
- detectFormatterCommand currently returns npm run format whenever package.json has scripts.format, which this pnpm repo does. install also always uses npm install.
- Fix package manager detection to prefer pnpm-lock.yaml, then yarn.lock, then package-lock.json/npm. Use pnpm install --frozen-lockfile and pnpm run format for pnpm repos. This matters because npm install can create package-lock.json.

4. getPullRequest failure returns executed:false repair_failed without graph decision.
- If executor cannot load PR details, operators won't see a repair_failed decision and retry/budget accounting is weak.
- Prefer creating the worker run before getPullRequest, then record repair_failed decision on lookup failure. If you keep early return, explain why and add a graph event/decision another way. I prefer record decision.

5. Scheduler tests currently assert executor is called for active/terminal guards because the guard is inside executor. That is okay only if we explicitly treat runRepairerExecutor as the orchestration boundary. But the test names should reflect this, or better add executor tests proving active/terminal no-op. We need at least one assertion that no new worker_run/decision is created when active/terminal exists.

Please implement fixes, run the focused tests again, and summarize changed files/tests.
