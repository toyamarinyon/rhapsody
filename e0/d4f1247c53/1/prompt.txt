Title: Move post-PR decisions out of builder and add config-driven format repair detection
Body: ## Summary

Builder/runner should stop making post-PR decisions such as moving Project items to Human Review immediately after creating a PR. Its responsibility should end at branch push, PR creation, and artifact recording.

Post-PR observation should happen through the scheduler/post-PR worker path. After a PR is created, the Project item should remain In Progress while checks are pending, repairable failures are repaired, and final review/merge decisions are made.

Format repair detection should also become config-driven. GitHub CheckRun names are often too coarse, for example `Static checks`, while the actual failing Actions step may be `Format check`.

## Desired Behavior

- Builder creates or reuses the GitHub PR and records branch/PR artifacts.
- Builder does not move the Project item to Human Review after PR creation.
- Scheduler observes In Progress items with PR artifacts.
- `checks_pending` keeps the item In Progress.
- `ci_failed + repair_allowed` runs the repairer and keeps the item In Progress.
- `ci_failed + repair_blocked` moves the item to Human Review.
- `checks_success` applies the existing post-run policy: auto-merge when eligible, otherwise Human Review.

## Repair Config

Add support for explicit format repair matching in `.rhapsody/config.toml`:

```toml
[repair]

[[repair.format_checks]]
workflow_path = ".github/workflows/ci.yml"
job_name = "Static checks"
step_names = ["Format check"]
```

`step_names` means: if one of these failed Actions steps is present for the configured workflow/job, the failure may be classified as format-fixable.

Do not add `repair_command` yet. The existing repair wrapper already detects the package manager and `format` script.

## Implementation Notes

- Remove post-run decision/action side effects from `runnerWorkflow`.
- Keep PR handoff and builder outcome recording in `runnerWorkflow`.
- Move Human Review / Done / auto-merge decisions into the scheduler post-PR path.
- Extend GitHub check summary data with Actions workflow/job/failed step metadata.
- Prefer config-based repair matching when configured.
- Keep the existing CheckRun-name heuristic as a fallback when no repair config exists or Actions step metadata is unavailable.

## Acceptance Criteria

- A newly created PR leaves the Project item In Progress.
- A pending CI result does not move the item to Human Review.
- A failed `Static checks` job with failed step `Format check` matches the configured format repair rule.
- A non-matching failed step does not get classified as format-fixable.
- A repairable failure starts the repairer without changing the item out of In Progress.
- A non-repairable failed check moves the item to Human Review.
- Passing checks still lead to auto-merge or Human Review according to existing post-run policy.
